Sprint Review

In this sprint I successfully added three new functions. I separated my main function into the main function and the order function, I also added an adding to the pizza quantity function and I added a remove pizza quantity function. The main function now just has a basic menu which asks the user if they want delivery or pick up and redirects the user's choice to the function that they selected. From there, the delivery or pick up function then redirects to the order function.

I will add more to the delivery and pick up functions in my next sprint, so that the program asks for the address for and phone number for the delivery option and asks for the phone number in the pick up option.

In the order function, it has a menu which asks the user if they want to add pizza quantity to the order, remove from the pizza quantity or quit. If the user chooses to add to the pizza quantity, it redirects them to the add_pizza function.

The add_pizza function prints out the menu and then asks the user what pizza they want to add. The function then adds to the pizza quantity depending on the amount the user selects and updates the cost. If then asks the user if they want to add another pizza and if they do, the add_pizza function runs again, if they don't the program returns to the order menu, where the user can then choose to add another pizza again, remove pizzas or quit. In a later sprint, I will set restrictions on how many pizzas the user can order.

The remove pizza function allows user to select a pizza type from the list and choose the quantity of pizza that they want to remove. However, an issue I have with this is that the user is able to go into negative quantities. I want to make the program to only allow the user to reduce the amount that is present and I will likely fix this in my next sprint or when I do validation.


While I was creating sprint 2 of the program I had to run quite a few tests as I had issues when I was designing the order function. As I said before, I split the main function into the main function and an order function. One of the reasons I did this was because I had a lot of repetition in my delivery and pick up functions. As they both do the same thing for the main part of the pizza program (which was what I was designing in sprint 2), it made sense to make an order function and then redirect the delivery and pick up function to it. I also decided to do this for my quit option in the add and remove functions, it needed somewhere to redirect to after the user was done and I didn't want it repeating the delivery or pick up messages and I also didn't want it to redirect to the main menu.

Then, I had an issue with the quitting option in the order function. First, instead of stopping the program, it would redirect to the start of the order function. Then, I changed something and it would redirect to the main menu and the user would have to press quit again for the program to stop. This design was unconventional and I had to run some tests to figure out where to put/remove loops and I figured out how to allow the program to quit at the order function.

For my next sprint, I want to add a function that allows the user to add new pizza types to the menu list. I also want to add a function that allows users to delete a pizza from the menu/order. As well as this, I plan on adding a review/confirm function where the user can see their order and confirm it. After they confirm their order a new order should start. I also want to add a function that just prints out the menu.

Sprint Review


In my 3rd sprint, I decided to do change the base structure of my pizza program. In my previous sprint, I had the order details stored within a list called "pizza_list", where when a pizza is added to the order, the quantity would update within the pizza list and the cost would be set to $5 within the list. However, this design was not flexible and also caused some problems when I wanted to further extend my program, so I had to change it. The new structure works well, however.

One of the problems it caused was that when I was creating my review function, I couldn't print out the customer's order without printing out all of the pizza list options (so it would print pizza options where the quantity was 0 as well as pizza options that had a quantity more than 0). Visually, this was not appealing and it didn't make sense as a program to print out what the user had not ordered. So, I decided to instead use a list to store the pizza menu options, an empty list to store customer order details.

In the empty customer order list, it allowed me to add what the user ordered and the list kept updating as they ordered more or less. As the list only contained what the user ordered, it meant that I was able to only print what they ordered in the review function.


With the new program, the new pizza function is used to add pizzas to the order and then the add pizza function allows the user to add to the quantity of pizzas for pizza type that is already in the order list. For my next sprint, I want to combine the add, remove and delete pizza type functions in a menu option called "edit order". This would make the main menu less confusing and easier to read. In the "edit order" option I want to present the options of adding or removing to the user. Then, the user will be shown the order list and can add or remove pizza quantities.

I also removed the order function. The order function was excessive and unnecessary as I could include everything in the order function in the main function. It was also confusing to work with when I was creating exit options as it had to go through the main function to quit.

In sprint 3, as mentioned earlier, I added a review function. The review function allows users to see what the order consists of and review the customer details such as deliver address, name, etc. In the review function, it asks the user if they wish to confirm the order or if they want to continue adding more items to the order. It also prints out the customer details. I will update this in my next sprint to make the review function more thorough, however. I want to rearrange the format so that the information given is clearer.

I also changed the order of operations slightly so that the program runs more smoothly. I now have the delivery/pick up function as an option in the menu instead of starting the program with the deliver/pick up option. I am also going to have the program not start with adding a new pizza immediately and instead give the user a menu of options. To make sure the order only goes through if a pizza is added to the order, I am going to validate my program so that the review function cannot be called if nothing has been added to the order. I will also validate so that if the user enters a pizza that is already in their order, it redirects to the add pizza function instead of entering a value twice. I will also validate integer entries so that the program will not crash if a value that is not an integer is entered and I need to validate for string entries so that there is maximum and minimum lengths. I need to validate for string lengths to minimise the chances of invalid/fake names will be entered.

I am also going to have the delivery/pick up function repeat the details the user entered as a confirmation message to check that the details are correct and give the user the option to change the details if they are wrong. I also need to validate the phone number so that the program only accepts phone numbers with a certain length and prefix.

Relevant Implications Summary

Future proofing

Future proofing is where you design your program to be easily updatable in the future. This means that the program is easy to add new features to and changes to different parts of the program are easy to change.

In my program I have addressed future proofing by using a function based build. Each of the different features in my program have a different function. So, for example I have a function specifically for adding pizzas to the order as well as a function specifically for removing pizzas. By using different functions for different features, I am able to update and make changes to a function without "upsetting" the rest of the program. This means that if something goes wrong with a function, it is unlikely to disrupt the entire program. It also means it is easier for me to fix errors as I can target the error to a function, take the function out of the main program and make tests with it, without disrupting the main program.

My program is also easy for another developer to work on and improve. My code is well commented and has docstrings at the top of every function. The docstrings have a description of what the function does and they also say what the parameters refer to in the program. This makes it easier for the developer to see what each function does. As my program starts with the main function, it also makes it easy for a developer to see how my program works and make changes.

I also used a python 3 code checker to see if my code followed the main code conventions. The code checker told me how to write proper docstrings, proper formatting for my program and it also told me when I had lines of code which were too long. The code checker brought up a good point about line length being too long. It said that lines of code longer than 79 characters should be shortened or distributed across multiple lines. The reason they suggested this was because shorter means the lines of code are likely to be less complicated and easier to understand by other people. They also mentioned that it is also easier to work using files side by side with shorter lines of code. As well as this, they also said that shorter lines of code work well for code review tools that show the information in two side by side columns. As shown in my "Style Checker" section in this website, I made sure to have no style issues left in my code after going through the code checker.

Functionality

Functionality is making sure the program works well and runs smoothly. This means that the program does not crash, totals get added properly and pizzas are added properly.

In my program, I have addressed functionality by systematically testing my program after each sprint, as well as regularly testing my program throughout the addition of a new feature. As I have thoroughly tested my program, I have been able to spot most (if not all) errors and fixed them so that the program runs smoothly. As far as I am concerned, my program does not crash, so the user should not have any serious issues whilst using the program.

Through thorough testing as well, I have made sure that the total cost and quantity of pizzas adds up to the correct amount throughout the program. I calculate the total cost and total quantity individually through each function, instead of having a running total throughout. Each cost and quantity is also printed out as well as the total cost. By doing this, it is clear to see where an error may be in calculations while I was testing my program, so any errors were easy to fix. The quantity of pizzas is a value that gets updated throughout the program, however I have tested this regularly throughout all my sprints to make sure that the correct amount is removed or added to.

As well as my personal testing, I have also had multiple other people test my program, some with digital technologies experience and some with none. By testing my program with people who had digital technologies knowledge, it helped me to spot and fix technical errors with my program that I may have not noticed. Testing my program with people that were not knowledgeable with digital technologies was also helpful as they told me what parts of the program were easy to understand and what parts were slightly confusing. An example is where my friend told me that the integer error message was confusing so I changed it to make it more understandable.

Usability

Usability is where the program is easy to use for the user. This means tha the program gives informative feedback, communicates if the user made an error and if applicable, fixes the error on its own.

In my program I have given informative user feedback through the use of messages that appear after a function has completed its task. An example of this is how I have a message telling the user how many pizzas were added in my adding pizza function before the user returns to the main menu. I have made my messages so that they are not "computer talk" and they are easy for a person without digital technologies knowledge to understand.

I also have error messages every time the user enters a value that is not accepted by the program. When the user enters a wrong value, a brief message stating what the user needs to input appears until they enter an acceptable value.

For input where the user has to enter (Y) or (N) etc I also have validated so that the program accepts both upper and lower case letters as it can be frustrating having to type in upper case for the whole program. Another self fix is how the program ignores spaces if they are typed before the letter when the user enters a menu option or a (Y)/(N) entry.

My program also uses a consistent navigation system. It is similarly designed to common shopping programs that people use daily, it is very predictable and easy to learn how to use. The user is also able to return to the main menu easily if they get lost and once they are at the main menu they can quit at any time.
