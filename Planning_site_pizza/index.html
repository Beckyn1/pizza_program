Sprint Review

For sprint 1, I wanted to create a basic pizza menu which had two options available on it. I also wanted the program to ask the user if they wanted delivery or pick up and made the program ask for the user's name. As well as this, I also got the program to print out the list of pizza options available, ask the user what they wanted and repeated what they chose. This will be the basic structure for my program, though for my next sprint, I need to work out how I should create the ordering part of the program. For my ordering program, it needs to be able to connect to both the pizza menu list and the options list for what the user wants to do. For this, I will do some testing to see if the ordering part would work better as a part of the main function or as it's own separate function. After I figure out how I want to design the order system, I want to add functions that allow the user to add the number of pizzas the customer ordered to the order and also take away amounts. I also want to have the cost of the order update depending on how many pizzas are ordered. I also want to change the design of the delivery and pick up system, so that the code isn't as repetitive as it is currently the same code in both functions except one is for delivery and one is for pick up and it seems unconventional. I am happy with how my first sprint has ran and I think it suits the basic purpose fine, though there are definitely a lot of things I still need to fix.

Sprint Review

In this version I added three new functions. I separated my main function into the main function and the order function, I also added an adding to the pizza quantity function and I added a remove pizza quantity function. The main function now just has a basic menu which asks the user if they want delivery or pick up and redirects the user's choice to the function that they selected. From there, the delivery or pick up function then redirects to the order function. I will add more to the delivery and pick up functions in my next sprint, so that the program asks for the address and phone number when the user chooses the pick up option. In the order function, it has a menu which asks the user if they want to add pizza quantity to the order, remove from the pizza quantity or quit. If the user chooses to add to the pizza quantity, it redirects them to the add_pizza function. The add_pizza function prints out the menu and then asks the user what pizza they want to add. Currently the user has to select the item menu number but for the future I want to look into changing this so that they can select the menu number or the word version (eg. cheese pizza instead of 0). The function then adds to the pizza quantity depending on the amount the user selects and updates the cost. If then asks the user if they want to add another pizza and if they do, the add_pizza function runs again, if they don't the program returns to the order menu, where the user can then choose to add another pizza again, remove pizzas or quit. The remove pizza function allows user to select a pizza type from the list and choose the quantity of pizza that they want to remove. However, an issue I have with this is that the user is able to go into negative quantities. I want to make the program to only allow the user to reduce the amount that is present and I will likely fix this in my next sprint or when I do validation. Another issue with the remove function is how it subtracts from the cost. I want the program to eventually add to a total cost as a customer orders more pizza and then have the cost removed from the total amount instead of just saying - $10 as it currently does. I will likely fix this when I reach sprint 4 however. Another issue that I need to fix is my messages and formatting. Some of the messages and formatting are hard to read and are confusing. I will fix this in my final sprint or when I do validation however.


While I was creating sprint 2 of the program I had to run quite a few tests as I had issues when I was designing the order function. As I said before, I split the main function into the main function and an order function. One of the reasons I did this was because I had a lot of repetition in my delivery and pick up functions. In my old design, they had the order code but kept inside delivery and pick up functions (basically the order function was displayed twice). As they both do the same thing for the main part of the pizza program (which was what I was designing in sprint 2), it made sense to make an order function and then redirect the delivery and pick up function to it. I also decided to do this because for my quit option in the add and remove functions, it needed somewhere to redirect to after the user was done and I didn't want it repeating the delivery or pick up messages and I also didn't want it to redirect to the main menu. Then, I had an issue with the quitting option in the order function. First, instead of stopping the program, it would redirect to the start of the order function. Then, I changed something and it would redirect to the main menu and the user would have to press quit again for the program to stop. This design was unconventional and I had to run some tests to figure out where to put/remove loops and I figured out how to allow the program to quit at the order function. I think the order design is quite good now however, as it is easy to update and add new functions to. When I come to making a confirmation or review function that prints out the full order it should be quite a simple process.
